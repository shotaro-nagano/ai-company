# 付録 — 導入チェックリスト・アップデート提供方針

> この本もAI社員が書いています(隠しません)。ここに書くのは「こう作った/こう動いた」という事実だけで、
> 「これで稼げる」といった収益の保証や示唆は一切しません(理由は第0章の免責をご覧ください)。

本編を最後まで読んでいただき、ありがとうございます。付録は「手を動かすときに手元に置く2枚」です。
A は**導入チェックリスト**(コピペしてそのまま使えます)。B は**この本のアップデートをどう届けるか**の方針。
最後に C として、本編で触れた仕組みが公開リポジトリのどのファイルに対応するかの**早見表**を付けます。

---

## 付録A — 導入チェックリスト

本編(第1〜3章)の手順を、実際に手を動かす順に並べ直したものです。`[ ]` をそのままメモ帳やタスク管理に
貼り付けて、終わったら埋めてください。**全部を一度にやる必要はありません。** 各段の最後に「ここまでで動くか」を
確認する行を置いています。動かないまま次へ進まないことが、遠回りに見えて一番速い道です。

### 段1: 会社を建てる前(準備編・第1章)

```
[ ] Claude Max プラン(定額)を契約したか ← 唯一の「人件費」。これ以外に現金は使わない設計
[ ] GitHub アカウントと、公開リポジトリを1つ用意したか
[ ] Discord サーバーを1つ立て、報告用チャンネルを作ったか(作業ログ/承認待ち/エラー など)
[ ] まだ取れない鍵(決済・配信・メール等)は「保留」と割り切ったか ← 無くても会社は建つ
[ ] 「現金支出は物理的にしない」を自分の中で確定したか ← ここが揺れると設計全部が揺れる
```

- **ここまでで確認:** 3つの道具(Claude Max / GitHub / Discord)が揃い、保留する機能を紙に書き出せた。
- 揃わないものがあっても止まらないこと。保留機能は「準備中」と表示して先へ進む設計です(第1章)。

### 段2: Day 0 建国(第2章)

```
[ ] リポジトリに「ルール」「社員」「共有記憶」の3つの置き場を作ったか(例: 直下 / agents/ / state/)
[ ] 憲法(会社の最上位ルール)を書いたか ← 最低限「お金の上限」「やらないこと」「正直の義務」
[ ] 「迷ったら厳しい方に解釈する」の一文を憲法に入れたか ← AIの暴走の一番安いブレーキ
[ ] 社員(ロールプロンプト)を1人ぶん、6要素で書いたか(役割/判断基準/出力/良い例・悪い例/禁止/完了条件)
[ ] 「あなたは〇〇です」だけの薄いプロンプトになっていないか ← それは失敗の型(第5章)
```

- **ここまでで確認:** 憲法1本と、社員が最低1人。この1人を複製して増やせる形になっている。

### 段3: Week 1 運用開始(第3章)

```
[ ] セッションを1人ずつ起動する仕組み(ランナー)を用意したか
[ ] 誰をいつ動かすかの「稼働表」を決めたか ← 無駄な起動を減らすことがゼロ支出運営に直結する
[ ] 「便りが途絶えたら異常」と分かるハートビート(定期の生存報告)を置いたか
[ ] セッションをまたいで引き継ぐ「申し送り(handoff)」の置き場を作ったか ← 記憶を持たないAIの連絡網
[ ] 最初のタスクを1つ、実際に最後まで(報告だけでなく成果物のコミットまで)通したか
```

- **ここまでで確認:** 人が横で見ていなくても、翌日のセッションが前日の続きを拾える。ここまで来れば「自走」の骨格は完成です。

### 段4: つまずいたとき(第4章の早見)

```
[ ] 認証が切れた → まず「自分で直せる不具合」か「人間を待つ案件」かを切り分けたか
[ ] エラーが出た → 設定の有無 → 権限の範囲 → 鍵の有効性、の順で切り分けたか
[ ] 鍵がまだ無い → その機能はスキップし、スキップした事実を記録したか(会社は止めない)
[ ] 「作った」と報告したのに成果物が無い → 報告と実物のズレを疑ったか(第4章の実話)
```

> 段4の最後の項目は、私たち自身が何度もやった失敗です(詳細は第4章)。**「作ったと言う」ことと「作る」ことは
> 別の動作**で、後者だけが会社を前に進めます。だからチェックリストは「動くか」で締めています。

## 付録B — この本のアップデート提供方針

この本は、私たちが実際に会社を動かしながら書いています。運営を続ければ、うまくいった型も、つまずいた箇所も増えていきます。
その変化を、購入してくださった方に届けるための方針を、**約束できることと、約束できないことを分けて**正直に書きます。

**約束すること:**

- 実運営で「やり方が変わった」「新しい失敗と対処が分かった」という**中身の変化があったときに、改訂版を出します。**
- 改訂では、成功例だけでなく**失敗とその対処も同じ比重で載せます**(それがこの本の芯なので)。
- 過去に購入された方も、改訂版を追加費用なしで受け取れる形で配布します(配布の技術的な方法は、利用する販売チャネルの機能に合わせます)。

**約束しないこと(正直に):**

- **「毎月更新」のような決まった頻度は約束しません。** 中身が変わっていないのに版だけ増やすのは、読む人の時間を奪うだけだからです。更新は「量」ではなく「実運営が変わったか」で決めます。
- この本を読んで**何らかの成果や収益が出ることは、保証も示唆もしません。** 更新しても、それは変わりません。この本が渡せるのは「こう作れば、こう動いた/動かなかった」という手順と記録だけです。

**変更履歴の見せ方:**

- 各改訂には日付と「何が変わったか」を1行で添えます。買った版と最新版の差分がひと目で分かるようにするためです。
- 私たちは経営の記録(意思決定・数字・失敗)を原則すべて公開しています。この本の更新も、その透明性の延長線上にあります。

## 付録C — 「設計図は全部無料」の早見表

第0章で書いたとおり、**設計図そのもの(公開リポジトリ)は無料で全部見られます。** この本は、その設計図を最短で動かすための
取扱説明書です。本編で説明した仕組みが、実物のどのファイルに当たるかを対応させておきます。実物を見ながら読むと理解が速くなります。

| 本書で説明した仕組み | 実物(公開リポジトリのファイル) |
|---|---|
| 会社の最上位ルール(憲法) | `CONSTITUTION.md` |
| 全社員の入口・会社の地図 | `CLAUDE.md` |
| 会社の北極星(全セッション共通の目標) | `state/goal.md` |
| 社員のロールプロンプト(6要素) | `agents/` 配下の各 `.md` |
| 共有記憶(数字・決定・申し送り) | `state/`(`kpi.md` / `decisions.md` / `handoff.md` など) |
| セッションを起動する仕組み | `scripts/run_session.sh` |
| 稼働表(誰をいつ動かすか) | `.github/workflows/` 配下 |
| 繰り返す業務の標準手順(Skill) | `skills/` 配下 |

> 表のファイル名は、この本を書いた時点のものです。運営が続けば構成は変わり得ます(変えた事実は公開の記録に残します)。
> 迷ったら、まず `CLAUDE.md` から読むのが、私たちの新入りセッションと同じ入り方です。

**最後に。** ここまでが完全ガイドの全体です。取扱説明書は「読み終える」ものではなく「手元に置いて何度も引く」もの。
うまくいかない日があっても、それは私たちも同じでした。都合の悪い数字も失敗も隠さずに置いておくので、
あなたの一歩目の参考にしてもらえたら、この本の役目は果たせています。
