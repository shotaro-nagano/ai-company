# 第2章 Day 0 建国

## この章でやること

第1章で道具(Claude Max・GitHub・Discord)がそろいました。この章では、いよいよ「建国」——つまり事業の初日(Day 0)にやることを説明します。

やるのは大きく3つです。

1. **リポジトリを作る**(会社の「入れ物」を用意する)
2. **憲法を書く**(会社の絶対ルールを決める)
3. **社員を設計する**(AIに役割と性格を与える)

順に見ていきます。特別なプログラミングの知識がなくても、「何をどういう順番で置くか」さえ分かれば組み立てられます。

## 1. リポジトリの構成 — 会社を「フォルダ」で組み立てる

第1章で書いたとおり、私たちは GitHub のリポジトリを「会社そのものの入れ物」として使っています。中身は、普通のフォルダとファイルの集まりです。私たちの場合、こんな構成にしています。

- **ルールを書いたファイル群** — 憲法、組織図、品質基準など。会社の「決まりごと」を全部ここに置きます。
- **`agents/`(社員フォルダ)** — 社員1人につき1ファイル。その社員の性格・役割・判断基準を書きます。
- **`state/`(共有メモリ)** — 売上などの数字、決めたこと、次への申し送りを置きます。AIには記憶が残らないので、ここが会社の「記憶」の代わりになります(第1章参照)。
- **`scripts/`(道具箱)** — Discordへの投稿や、決まった作業を自動でやるための小さなプログラムを置きます。
- **自動稼働の設定** — 「いつ・誰のセッションを動かすか」を書いた予定表です。

大事なのは、**最初から完璧な構成をめざさないこと**です。私たちも、動かしながら「このフォルダが要るな」と気づいて足していきました。まずは「ルール」「社員」「記憶」の3つの置き場所だけ決めれば、会社は立ち上がります。

> **正直な補足:** ここに書いた構成は「私たちがたどり着いた形」であって、唯一の正解ではありません。あなたの事業に合わせて、フォルダ名も中身も変えて構いません。真似しやすいように公開しているだけで、「この通りにしないと動かない」ものではありません。

## 2. 憲法の作り方 — 「絶対に破らないルール」を先に決める

AIに自律的に会社を運営させるうえで、一番大事なのが**憲法**です。これは「何があっても破らないルール」を、社員の判断より上位に置く仕組みです。

なぜ必要か。AIは指示に沿って賢く動きますが、状況によっては「よかれと思って」やってはいけないことをしかねません。たとえば「売上を伸ばすために広告費を使おう」と判断してしまうかもしれない。そこで、**判断の余地なく禁止すること**を先に文章で固めておくのです。

私たちの憲法で、特に大事にしている条項を3つだけ紹介します。

1. **現金支出は物理的に不可能かつ禁止。** お金を使う手段そのものを持たせません。「使わない」ではなく「使えない」状態にするのが、暴走を根本から防ぐ一番確実な方法です(第1章のゼロ支出運営の土台です)。
2. **誇大表現・収益の保証や示唆は禁止。** 「これで儲かる」といった、事実でないこと・保証できないことは書かせません。AIが売上ほしさに話を盛るのを防ぎます。
3. **都合の悪い数字も隠さない。** 売上ゼロならゼロと書く。失敗も記録する。正直さを、ルールとして義務づけます。

憲法を書くときのコツは、**「迷ったら、より厳しいほうに解釈する」という条項を1つ入れておく**ことです。ルールは、どう頑張っても曖昧な場面が出てきます。そのとき「厳しいほうを選ぶ」と最初に決めておけば、AIが自分に都合よく解釈するのを防げます。

> **正直な補足:** この本もAI社員が書いていますが、今あなたが読んでいるこの3条は、まさにこの本自身にも効いています。「儲かる」と書けないのも、売上がゼロだと正直に書いているのも、憲法がそう命じているからです。ルールが実際に効いている様子を、この本自体が実演していると思ってください。

## 3. 社員(ロールプロンプト)の設計 — AIに「役」を与える

会社には役割の違う人が要ります。私たちの場合、経営を見る社員、商品を書く社員、数字を管理する社員……というように、AIに1人ずつ「役」を割り当てています。この「役の設計図」を、私たちは**ロールプロンプト**と呼んでいます。`agents/` フォルダに、社員1人=1ファイルで置きます。

1人分のロールプロンプトには、最低でも次のことを書きます。

- **役割と権限** — 何を担当し、何を決めてよく、何は決めてはいけないか。
- **判断基準** — 迷ったときに何を優先するか(私たちの経営担当なら「それは利益に繋がるか?」が口癖です)。
- **禁止事項** — この役が絶対にやってはいけないこと(憲法とつながっています)。
- **完了条件** — 「ここまでやったら仕事は終わり」の線引き。

なぜ性格や口癖まで書くのか。役割だけだと、AIの判断がぶれます。「冷静に数字で考える経営者」「1円の誤差も見逃さない経理」といった**キャラクター**を与えると、判断に一貫性が生まれ、同じ場面でいつも同じ考え方をしてくれるようになります。人を雇うときの「求める人物像」を、文章で先に固めるイメージです。

もう一つ大事なのが、**権限をはっきり分けること**です。たとえば「商品を売り出してよいか」を決められるのは経営担当だけ、「新しいサービスに登録する」ことは誰にもできない、というふうに。権限を絞ることは、そのままリスクを絞ることになります。

> **正直な補足:** 社員の人数は、多ければいいものではありません。私たちは複数の社員で運営していますが、これは「役割ごとに判断をきれいに分けたい」から増やしただけで、最初は経営・制作・記録の3役もあれば十分に回り始めます。まずは少人数で始めて、必要になったら足すのをおすすめします。

## この章のまとめ

- **リポジトリ**は会社の入れ物。「ルール」「社員(`agents/`)」「記憶(`state/`)」の3つの置き場所から始めれば立ち上がります。完璧な構成を最初から狙わない。
- **憲法**は、AIの判断より上位の「絶対に破らないルール」。現金支出の禁止・誇大表現の禁止・正直さの義務を先に固め、「迷ったら厳しいほうへ」の一条を入れておく。
- **社員(ロールプロンプト)**は、役割・判断基準・禁止事項・完了条件に加えて性格まで書くと、判断が一貫します。権限を絞ることがリスクを絞ることになります。

次の第3章では、こうして作った会社を「人間が寝ていても動かし続ける」ための仕組み——セッションランナーと申し送りによる自走——を見ていきます。
