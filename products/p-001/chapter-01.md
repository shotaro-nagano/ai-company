# 第1章 準備編

## この章でやること

事業を立ち上げる前に、何を用意すればいいのかを整理します。結論から言うと、必要なものはとても少なく、しかも**追加の現金支出はゼロ**で始められます。私たち自身が実際にそれで動いています。

用意するのは、大きく3つだけです。

1. **Claude Max**(AI社員を動かす頭脳)
2. **GitHub**(会社の記憶と自動稼働の土台)
3. **Discord**(社員同士・人間との連絡窓口)

順に、それが何のために要るのかを説明します。

## 1. Claude Max — AI社員を動かす頭脳

この事業のすべての「社員」は、Claude(Anthropic の AI)です。企画も、執筆も、監修も、経営判断も、Claude が担います。

私たちが使っているのは **Claude Max** の定額プランです。ここが大事な設計思想なのですが、私たちは「使った分だけ課金される従量制」ではなく、**定額プランの枠内だけで全社を動かす**というルールにしています。理由は単純で、AIだけで自律運営させると、気づかないうちに使いすぎてコストが膨らむリスクがあるからです。定額にしておけば、月々の費用は「最初から決まった額」で固定できます。

そのため私たちの会社には「現金支出は物理的に不可能かつ禁止」という憲法条項があります(詳しくは第2章)。新しい有料サービスに登録することも、追加課金することもしません。Claude Max の定額の中だけで、全部やりくりします。

> **正直な補足:** 定額枠にも上限があります。私たちは使用量を常に監視する担当(社員)を置き、月間枠の80%を超えたら省エネ運転、85%を超えたら必須業務以外を止める、というルールで運用しています。「無制限に使える」わけではない、というのは先に書いておきます。

## 2. GitHub — 会社の記憶と自動稼働の土台

GitHub は、プログラムのソースコードを保管・共有するためのサービスですが、私たちはこれを**「会社そのものの入れ物」**として使っています。

具体的には、GitHub のリポジトリ(=ファイルの保管庫)の中に、次のものが全部入っています。

- **憲法・規程**(会社のルール)
- **社員のロールプロンプト**(各AI社員の性格・役割・判断基準を書いたファイル)
- **共有メモリ**(売上などの数字、決定事項、次への申し送り)
- **自動稼働の設定**(いつ・誰のセッションを起動するか)

AI社員には「記憶」が残りません。セッションが終われば、その回に考えたことは消えます。そこで私たちは、**大事なことは全部ファイルに書いてリポジトリに残す**という方法をとっています。次に起動した社員は、まずそのファイルを読んで「前回までに何があったか」を把握してから仕事を始めます。これが「人間が寝ていても会社が進む」仕組みの中心です(詳しくは第3章)。

GitHub には無料で使える範囲があり、私たちはその範囲で運用しています。ここでも追加の現金支出はありません。

## 3. Discord — 社員同士・人間との連絡窓口

Discord は、チャットのサービスです。私たちはこれを**社内の連絡ボード**として使っています。

- 各社員が仕事の報告を書き込む「作業ログ」
- 人間のオーナーが指示を出す「社長室」
- 承認が必要なことだけを掲示する「承認待ち」

AI社員は、セッションのたびにこれらのチャンネルを読み、必要なら書き込みます。人間のオーナーは、Discord を見れば「今どの社員が何をしているか」が分かり、指示があれば書き込むだけで伝わります。

Discord も無料の範囲で使えます。

## ゼロ支出運営の原則

ここまでで気づかれたかもしれませんが、3つの道具はどれも「定額または無料の範囲」で収まっています。これは偶然ではなく、**意図的な設計**です。

私たちの会社の一番大事なルールの一つが「**追加の現金支出はしない**」です。理由は3つあります。

1. **AIに自由にお金を使わせるのは危険だから。** 判断ミスや暴走で、気づかないうちに出費が膨らむ事故を根本から防ぎます。「使えないようにしておく」のが一番安全です。
2. **本当に利益が出ているかが分かりやすいから。** 費用が「定額のClaude代」だけなら、売上からそれを引くだけで純利益が計算できます。ごまかしようがありません。
3. **誰でも真似できるから。** 特別な資金がなくても、同じ3つの道具さえあれば始められます。

もし将来「どうしてもお金を使う必要がある」と判断した場合、AI社員が勝手に決めることはできません。必ず人間のオーナーに「これこれの理由で、いくら使いたい」と掲示して、承認を待ちます。承認がなければ、使いません。

## 「まだ鍵がないもの」は、あと回しにする設計

現実には、始めた時点で全部の道具がそろわないことがあります。私たち自身がまさにそうでした。

たとえば、商品を売るには決済サービス(私たちの場合はStripe)が必要ですが、これは審査があり、通るまで使えません。ファイルを配布する場所(itch.io)や、サポート用のメール受信箱(Gmail)も、設定(=「鍵」)がそろうまでは動かせません。

ここで大事なのが、**「鍵がまだ無い機能は、無いなりに止めずに進む」**という考え方です。私たちはこう設計しています。

- 鍵がまだ取得できていない機能は、**その部分だけをスキップ**する。
- スキップしたことを**記録に残す**(あとで鍵がそろったら再開できるように)。
- 鍵が必要な作業以外は、**止めずに全部進める**。

たとえば私たちの場合、この本を書いている時点で決済の審査がまだ通っていません。だからといって「売れないなら何もしない」ではなく、**商品(この本)の制作を先に進めておき、決済の鍵が届いたらすぐ発売できる状態にしておく**、という進め方をしています。

> **正直な記録:** この本を書いている時点で、私たちの売上はゼロです。決済サービスの審査が通っていないため、まだ1本も売れる状態にありません。それでも制作を止めないのは、「鍵が届いた瞬間に動けるように準備しておく」ことが、待っている間にできる一番価値のある仕事だからです。うまくいく保証があるからではありません。

## この章のまとめ

- 必要なものは **Claude Max・GitHub・Discord** の3つだけ。追加の現金支出はゼロで始められます。
- **費用を定額に固定し、AIに追加の出費をさせない**のが、安全とごまかしのなさの両方を支える土台です。
- **鍵がまだ無い機能はスキップして記録し、それ以外は止めずに進める。** これが、そろっていない状態でも前に進むためのコツです。
