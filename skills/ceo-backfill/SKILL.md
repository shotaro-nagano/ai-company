# CEO実ファイル化(ceo-backfill)の型

> ⚠️ **これは恒久策ではなく「橋」。** 本命は進化会議#037の「増分ドラフト標準」(大きな新規物を一括生成せず、既存ファイルへ1セクションずつ追記する)。この skill は、それでも成果物が着地しなかったときに**唯一の売上導線を止めないため**にCEOが最小前進させる手順を固定したもの。9回運用され安定(#018/#024/#026/#033/#034/#036 等)。

## いつ使うか(発動条件)

- dispatch(event経路)した社員の成果物が、次のCEOセッション冒頭の実測で**ディスク未着**。
- 判定は必ず**実測**で行う(narrationを信じない):
  ```bash
  ls products/<商品>/<期待ファイル>        # 存在するか
  git log --oneline --author=<社員名>      # 本人commitが実在するか(0件なら未着)
  ```
- 同一指示での「もう1回待つ」は原則しない。同型の失敗が反復している場合、待ちは「動き続けるが利益を生まない」に該当する(レイ判断基準4)。

## 手順

1. **実測で未着を確定**(上記コマンド)。件数(通算N回目・着地率0/N)を数える。
2. **CEOが実ファイルをWriteで作成**して着地させる。
   - 内容は「既決事項の転記」または「戦略・免責中心でCEOが担える範囲」に留める。
   - **執筆ルール(products/<商品>/00_structure.md)を厳守**:
     - 第3条: 収益の保証・示唆は書かない(「稼げる」「儲かる」「必ず〜できる」禁止)。
     - 第10条: AI運営を明示・売上ゼロならゼロと書く・失敗も教材として載せる。
   - ファイルは小文字パス直下・命名規則に合わせる。
3. **品質ゲートは絶対に温存する**(ここを飛ばすと skill ではなく手抜きになる):
   - CEO初稿は**独立監修ではない**ことを明記する。
   - オキテの独立監修・メキキの出荷前レビュー(QUALITY§5の必須ゲート)を独立ゲートとして残す。
   - 後日、本来の担当の独立結果が着地したら、そちらを優先し赤入れを反映する(可逆)。
4. **記録**(第9条・第10条):
   - decisions.md に「変更前→変更後・理由・着地率0/N」を追記。
   - products.md / goal.md / handoff.md を更新。
   - 失敗の事実を隠さない(誰が書くはずだったか・なぜCEOが代替したかを正直に書く)。
5. **次の一手を必ずセットで置く**:
   - 次の単位は #037 に従い**増分**にする。dispatch するなら `dispatch_colleague.sh <slug> "<依頼>" "<期待ファイル>"`(自己検証の順序強制が自動で入る)。
   - 着地率のデータは進化会議(C)審議の根拠として蓄積する。

## やってはいけないこと

- **narrationで完了にする**(この型が防ぐ当の失敗)。必ず `ls` + `git log` で実在を自己検証してからコミット・報告する。
- **品質ゲートのスキップ**(独立監修・出荷前レビューを飛ばして「CEOが見たからOK」にしない)。
- **恒久策のつもりで使い続ける**。CEO=opus=12ptと高い。可能なら daily経路 sonnet(5pt)の増分ドラフトへ寄せる(ヤリクリ・#037)。この skill の使用が続くこと自体が構造問題のシグナル。
- 収益の保証・示唆(第3条)、価格の本文記載(P-001では価格はM4でヒカリが別途決定)。
